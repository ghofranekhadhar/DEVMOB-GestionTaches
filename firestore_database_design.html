👤 2. Collection users

📁 Collection racine
/users/{userId}

ID du document = Firebase Auth UID (pas d'auto-génération)

Champ Type Requis Défaut Description
email string Requis — Email de l'utilisateur (unique)
displayName string Optionnel null Nom d'affichage depuis Firebase Auth
fullName string Optionnel null Nom complet saisi par l'utilisateur
photoUrl string Optionnel null URL de la photo de profil
role string Requis "member" "admin" ou "member"
createdAt timestamp Auto serverTimestamp() Date de création du compte

Exemple de document

// users/abc123
{
  "email": "ahmed@example.com",
  "displayName": "Ahmed",
  "fullName": "Ahmed Ben Ali",
  "photoUrl": "https://...",
  "role": "member",
  "createdAt": "2026-04-01T14:00:00Z"
}
⚠️ Important — Sécurité du rôle
Le champ role ne doit jamais être modifiable depuis le client. Seul un admin peut le changer via la console Firebase ou une Cloud Function sécurisée.

📁 3. Collection projects

📁 Collection racine
/projects/{projectId}

ID du document = auto-généré par Firestore

Champ Type Requis Défaut Description
name string Requis — Nom du projet
description string Optionnel "" Description du projet
deadline timestamp Requis — Date limite du projet
members array<string> Requis [] Liste des UIDs des membres
createdBy string Requis — UID du créateur
status string Requis "en_cours" "en_cours" | "termine" | "en_pause"
completionPercentage number Auto 0 Calculé : (tâches done ÷ total) × 100
createdAt timestamp Auto serverTimestamp() Date de création
updatedAt timestamp Auto serverTimestamp() Dernière modification

Exemple de document

// projects/proj001
{
  "name": "Application Mobile v2",
  "description": "Refonte de l'application mobile",
  "deadline": "2026-06-15T00:00:00Z",
  "members": ["uid_ahmed", "uid_sara", "uid_khalil"],
  "createdBy": "uid_ahmed",
  "status": "en_cours",
  "completionPercentage": 45,
  "createdAt": "2026-04-01T10:00:00Z",
  "updatedAt": "2026-04-01T14:30:00Z"
}
ℹ️ Pourquoi members est un tableau et non une sous-collection ?
Avec 50-100 utilisateurs max, un tableau est plus performant : une seule lecture pour obtenir tous les membres, et Firestore supporte arrayContains pour les requêtes de filtrage. Une sous-collection serait préférable au-delà de ~500 membres.

✅ 4. Collection tasks

📁 Collection racine
/tasks/{taskId}

ID du document = auto-généré par Firestore

Champ Type Requis Défaut Description
title string Requis — Titre de la tâche
description string Optionnel "" Description détaillée
assignedTo string Optionnel "" UID de la personne assignée (vide = non assigné)
dueDate timestamp Requis — Date d'échéance
status string Requis "todo" "todo" | "in_progress" | "done"
projectId string Requis — ID du projet parent
projectName string Requis — Nom du projet (dénormalisé)
priority string Requis "medium" "low" | "medium" | "high"
comments array<map> Optionnel [] Commentaires embarqués (voir structure ci-dessous)
createdBy string Requis — UID du créateur de la tâche
createdAt timestamp Auto serverTimestamp() Date de création
updatedAt timestamp Auto serverTimestamp() Dernière modification

Structure d'un commentaire (embarqué dans comments[])

Champ Type Description
id string UUID généré côté client
userId string UID de l'auteur
userName string Nom de l'auteur (dénormalisé)
text string Contenu du commentaire
createdAt timestamp Date du commentaire

Exemple de document

// tasks/task001
{
  "title": "Implémenter l'écran de login",
  "description": "Créer la page de connexion avec validation",
  "assignedTo": "uid_sara",
  "dueDate": "2026-04-15T00:00:00Z",
  "status": "in_progress",
  "projectId": "proj001",
  "projectName": "Application Mobile v2",
  "priority": "high",
  "comments": [
    {
      "id": "c1",
      "userId": "uid_ahmed",
      "userName": "Ahmed",
      "text": "N'oublie pas la récupération du mot de passe",
      "createdAt": "2026-04-02T09:00:00Z"
    }
  ],
  "createdBy": "uid_ahmed",
  "createdAt": "2026-04-01T10:30:00Z",
  "updatedAt": "2026-04-02T09:00:00Z"
}
ℹ️ Pourquoi les tâches sont une collection racine ?
L'app doit afficher toutes les tâches d'un utilisateur (tous projets confondus) dans le Dashboard et "My Tasks". Avec une sous-collection, il faudrait une requête par projet. Ici, une seule requête where('assignedTo', '==', userId) suffit.
ℹ️ Pourquoi projectName est dénormalisé ?
Évite de charger le document project à chaque affichage de tâche. Quand un projet est renommé, il faut mettre à jour ce champ dans toutes ses tâches via un batch write.

🔔 5. Collection notifications

Champ Type Requis Défaut Description

Type Déclencheur

  "relatedProjectId": "proj001",
  "isRead": false,
  "createdAt": "2026-04-01T10:30:00Z"
}

📇 6. Index composites nécessaires

Firestore crée automatiquement les index simples. Les requêtes combinant plusieurs champs nécessitent des index composites :

Collection Champs indexés Cas d'usage
tasks assignedTo ↑, dueDate ↑ Mes tâches triées par échéance
tasks assignedTo ↑, status ↑ Mes tâches filtrées par statut
tasks projectId ↑, status ↑ Tâches d'un projet par statut
tasks projectId ↑, dueDate ↑ Tâches d'un projet par échéance
tasks assignedTo ↑, projectId ↑, status ↑ Mes tâches sur un projet, par statut
notifications userId ↑, isRead ↑, createdAt ↓ Notifications non-lues, plus récentes d'abord
notifications userId ↑, createdAt ↓ Toutes les notifications, triées
💡 Astuce
Vous n'avez pas besoin de créer ces index manuellement. Firestore affichera un lien dans la console de débogage Flutter quand une requête nécessite un index manquant. Cliquez dessus pour le créer automatiquement.

🔒 7. Règles de sécurité Firestore

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ─── Fonctions Helper ───
    function isAuthenticated() {
      return request.auth != null;
    }

    function isAdmin() {
      return isAuthenticated() &&
        get(/databases/$(database)/documents/users/
          $(request.auth.uid)).data.role == 'admin';
    }

    function isOwner(userId) {
      return isAuthenticated() &&
        request.auth.uid == userId;
    }

    // ─── USERS ───
    match /users/{userId} {
      allow read: if isAuthenticated();
      allow create: if isOwner(userId) &&
        request.resource.data.role == 'member';
      allow update: if isOwner(userId) &&
        request.resource.data.role == resource.data.role;
      allow delete: if isAdmin();
    }

    // ─── PROJECTS ───
    match /projects/{projectId} {
      allow read: if isAuthenticated();
      allow create: if isAdmin();
      allow update: if isAdmin() ||
        (isAuthenticated() &&
         request.auth.uid in resource.data.members);
      allow delete: if isAdmin();
    }

    // ─── TASKS ───
    match /tasks/{taskId} {
      allow read: if isAuthenticated();
      allow create: if isAdmin() ||
        (isAuthenticated() &&
         request.resource.data.createdBy == request.auth.uid);
      allow update: if isAdmin() ||
        (isAuthenticated() &&
         (resource.data.assignedTo == request.auth.uid ||
          resource.data.createdBy == request.auth.uid));
      allow delete: if isAdmin() ||
        (isAuthenticated() &&
         resource.data.createdBy == request.auth.uid);
    }

    // ─── NOTIFICATIONS ───
    match /notifications/{notifId} {
      allow read: if isAuthenticated() &&
        resource.data.userId == request.auth.uid;
      allow create: if isAuthenticated();
      allow update: if isAuthenticated() &&
        resource.data.userId == request.auth.uid;
      allow delete: if isAuthenticated() &&
        resource.data.userId == request.auth.uid;
    }
  }
}

🔍 8. Requêtes fréquentes (Dart / Flutter)

Dashboard — Tâches de l'utilisateur connecté

_firestore.collection('tasks')
  .where('assignedTo', isEqualTo: currentUserId)
  .orderBy('dueDate')
  .snapshots();

Dashboard — Projets de l'utilisateur

_firestore.collection('projects')
  .where('members', arrayContains: currentUserId)
  .snapshots();

Tâches d'un projet spécifique

_firestore.collection('tasks')
  .where('projectId', isEqualTo: projectId)
  .orderBy('status')
  .snapshots();

Calcul automatique du completionPercentage

Future<void> updateProjectCompletion(String projectId) async {
  final tasksSnapshot = await _firestore
    .collection('tasks')
    .where('projectId', isEqualTo: projectId)
    .get();

  final total = tasksSnapshot.docs.length;
  final done = tasksSnapshot.docs
    .where((doc) => doc.data()['status'] == 'done')
    .length;

  final percentage = total > 0
    ? ((done / total) * 100).round() : 0;

  await _firestore.collection('projects')
    .doc(projectId).update({
      'completionPercentage': percentage,
      'updatedAt': FieldValue.serverTimestamp(),
    });
}

Retirer un membre → désassigner ses tâches

Future<void> removeMember(String projectId, String userId) async {
  final batch = _firestore.batch();

  // 1. Retirer de la liste members
  batch.update(_firestore.collection('projects').doc(projectId), {
    'members': FieldValue.arrayRemove([userId]),
    'updatedAt': FieldValue.serverTimestamp(),
  });

  // 2. Désassigner ses tâches
  final tasks = await _firestore.collection('tasks')
    .where('projectId', isEqualTo: projectId)
    .where('assignedTo', isEqualTo: userId)
    .get();

  for (final doc in tasks.docs) {
    batch.update(doc.reference, {
      'assignedTo': '',
      'updatedAt': FieldValue.serverTimestamp(),
    });
  }

  await batch.commit();
}

Notifications non lues

_firestore.collection('notifications')
  .where('userId', isEqualTo: currentUserId)
  .where('isRead', isEqualTo: false)
  .orderBy('createdAt', descending: true)
  .snapshots();

🧠 9. Décisions de conception et justifications

Décision Justification
Tasks en collection racine (pas sous-collection de Projects) Permet de requêter toutes les tâches d'un utilisateur tous projets confondus, sans collection group query
Comments embarqués dans Tasks (array) Peu de commentaires attendus, toujours lus avec la tâche → évite des lectures supplémentaires
projectName dénormalisé dans Tasks Évite un get() sur le projet pour chaque tâche affichée en liste
members comme array dans Projects Volumétrie faible (<100 users), supporte arrayContains pour le filtrage
Notifications en collection séparée Permet le tri, filtrage (lues/non-lues), et suppression sans affecter les tâches
deadline / dueDate comme Timestamp Meilleure précision et tri natif Firestore (vs string ISO)
⚠️ Dénormalisation = responsabilité de synchronisation
Quand un projet est renommé → mettre à jour projectName dans toutes les tâches liées.
Quand un utilisateur change de nom → mettre à jour userName dans tous les commentaires.
Utilisez des batch writes pour garantir l'atomicité.

🔄 10. Différences avec le schéma actuel

Élément Actuellement Recommandé Raison
deadline / dueDate String Timestamp Tri et comparaisons natifs Firestore
createdAt / updatedAt Absent Timestamp Traçabilité et tri chronologique
createdBy dans tasks Absent String (UID) Savoir qui a créé la tâche (permissions)
Comments .user String (nom) userId + userName Permet de lier au vrai utilisateur
Comments .date String createdAt (Timestamp) Cohérence du schéma
Collection notifications Absente Nouvelle collection Support des notifications
